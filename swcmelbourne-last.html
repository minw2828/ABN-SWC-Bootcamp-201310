class DNASequence:
    def __init__(self, sequence):
        self.sequence = sequence
        self.base_counts = {}
    
    def base_count(self, base):
        if base in self.base_counts:
            return self.base_counts[base]
        else:
            count = self.sequence.count(base)
            self.base_counts[base] = count
            return count
    
    def gc_content(self):
       '''Given a sequence returns the fraction of the bases that are G or C.'''
        g = self.base_count('G')
        c = self.base_count('C')
        return float(g+c)/len(self.sequence)
        
    def reverse_complement(self):
        '''Given a sequence return the reverse complement of that sequence'''
        complements = {'G': 'C',

                       'T': 'A'}
        rev_c = ""
        for base in self.sequence:
            rev_c = complements[base] + rev_c
            
        return rev_c

Basic Git commands:

  • git init (will initialize a new Git repository inside a folder. It also affects all files and subfolders within. Only required once per project. Projects are managed at a folder level).
  • git status (will show you the current state of the repository. You'll use this all the time).
  • git show-branch (see recent commits from branches).
  • git checkout (switch branches, or grab files from earlier commits)
  • git add (stage files ready for committing)
  • git commit (commit files into Git's memory with a useful message)
  • git branch (list branches, create them or delete existing ones)
  • git push (sync changes with a remote repository)
  • git pull (sync changes from a remote to a local repository)
  • git merge (merge changes between branches)
  • git diff (show changes between current file and most recent commit).
  • git log (see detailed logs. Using the aliases defined above, you can see shorter one-line logs and also branch activity).

There are many more commands in Git but if you master this set it should serve you well for a long time.

#step 1: Generate a multiple alignment with muscle

muscle -in bombycilla_co1.fasta -out bombycilla_co1.fasta.aln

#step 2: convert the alignment format from FASTA alignment to Phylip format
python -c "import Bio.AlignIO; Bio.AlignIO.convert('bombycilla_co1.fasta.aln', 'fasta', 'bombycilla_co1.phy', 'phylip-relaxed')"

#step 3: Use RAxML to generate a tree from the Phylip format multiple alignment

raxmlHPC -m GTRCAT -n bombycilla_co1.phy -p 10000 -s bombycilla_co1.phy

#rename one of the output files, from RAxML, to newick extension
mv RAxML_result.bombycilla_co1.phy bombycilla_co1.new

# Step 4: Use a Python script to generate a graphic file (PNG) from the Newick format tree
python draw_tree.py bombycilla_co1.new bombycilla_co1.png

# Now lets take a look at how to automate this process using a Makefile
# Makefiles must have tab characters for the indentation. Spaces will NOT work.
target: dependencies
    commands
    
    
all: $(patsubst %.fasta, %.png, $(wildcard *.fasta))

# to retain other files generated by the build which are not targets or dependencies you need to use .PRECIOUS:
.PRECIOUS: %.fasta.aln %.phy %.new

clean:
    rm -f *.aln *.phy *.reduced *.new *.png

# $< : the input file
# $@ : the output file

# step 1: produce a FASTA alignment
%.fasta.aln: %.fasta
    muscle -in $< -out $@
    
# step 2: convert alignment from FASTA to Phylip
%.phy: %.fasta.aln
    python -c "import Bio.AlignIO; Bio.AlignIO.convert('$<', 'fasta', '$@', 'phylip-relaxed')"
    
# step 3: create a Newick tree using RAxML
%.new: %.phy
    rm -f RAxML_*
    raxmlHPC -m GTRCAT -n $< -p 10000 -s $<
    mv RAxML_result.$< $@
    
# step 4: generate an image
%.png: %.new draw_tree.py
    python draw_tree.py $< $@


'Make' syntax to generalise e.g. specific file names to general file names
%:   wildcard
$@ : the target
$< : the first dependency
$^:  all dependencies


# Clean up the intermediate files
rm -f *.aln *.phy *.reduced RAxML_* *.png *.new

If you run make with the -j <int> switch it will run the command multithreaded with <int> threads. So running our Makefile with -j 2 will run both command streams at once.
I've seen it with -j 64...

Can redirect output to log file:
make > log.txt

Collaborative Group Excersises - $50/person for the winning team
================================================================
Sickle ( https://github.com/najoshi/sickle ) is a windowed adaptive trimming tool for FASTQ files using quality.
As a team, you are tasked with developing a Python-based project implementing some of the features of Sickle. You will then look at the code developed by another team and try to improve their project. In both parts of the excercise you should demonstrate what you have learned over the last 2 days. This might include the use of:
  • Version control
  • Unit tests
  • Documentation
  • Automation using Make/Makefiles

Here's what to do:
1) Get together in groups of 3
2) One of you should setup the repository on github and add the other team members as collaborators so you all have commit privilages to that repo.
3) For the first 30min you're only allowed to discuss, write tests and documentation as well as write stub methods/classes. i.e. no real implementation please!
  a) Think about what features you'll try to implement in the short time you have. For example, do you want to just work on "quality" strings and develop code to do the windowed adaptive trimming on those string. Or do you want to go the whole hog and do FASTQ parsing and output? Will you restrict yourself to just one FASTQ encoding schema such as Phred+33 or Phred+64 or code support for all?
  b) Think about the documentation/tests a new team might need to find their way around your code and run it, test it etc.
  c) Decide how you might split the work between the team members
4) Before afternoon tea you should have pushed all your code to your github repository
5) Teahttps://github.com/bendmorris/make_tutorial.git
6) The person who setup your repository in step 2 above should perform a github "fork" of another teams project repository (we'll coordinate this below) and add the other members of the team as collaborators.
7) Try to understand what the code does by reading the documentation, running test etc.
8) Try to improve the project by contributing to the code, documentation and tests.
